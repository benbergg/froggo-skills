# 退款分析方法论与口径硬规则

> 本文件是 `refund-insight` skill 的"判断力沉淀"——踩过的坑、口径裁定、可复用方法。
> 数据层脚本已把这些固化为代码，本文件供深读与新客户调参时参考。

## 一、输入数据假设（班牛抖音售后导出）

明细 xlsx（Sheet1）预期含以下列（班牛工单系统导出格式）：

`工单编号(不可修改)、店铺、店铺ID、售后单号、申请时间、售后原因(平台)、售后原因(一级标签)、售后原因(二级标签)、售后原因(标签名称)、分析依据、图片依据、买家原声、买家情绪、进线总结、进线次数、商品名称、商品SKU、商品货号、售后状态、售后类型、退款金额、退款类型、买家留言、订单号、下单时间、买家昵称、创建人、创建时间、更新人、更新时间、聊天记录`

两套并行的退款原因（**核心认知**）：
- `售后原因(平台)`：顾客在抖音申请退款时**自选**的原因（覆盖率高，但约 9 成是"不想要了/7天无理由/多拍错拍"等**低信息**项）。
- `售后原因(标签名称)`：班牛 AI 从**聊天记录**推断的真实原因（仅"有进线"样本有效，其余为"非售后场景"）。

## 二、三个数据陷阱（必须处理）

1. **订单号/售后单号 float 精度损坏（硬伤）**
   xlsx 里 `订单号`/`售后单号` 以 float64 存储（19 位整数超出 float 15–16 位精度），**末 3–4 位是舍入垃圾值，不可用于回溯核实**。
   → 核实锚点一律用 **`工单编号`**（9 位整数，读为字符串保精度）。报告须声明此事。

2. **时间字段是字符串**
   `申请时间`/`下单时间` 是字符串而非日期型。解析后算 `退款间隔 = 申请 − 下单`；负值（异常）置空，不污染中位数。

3. **明细 ≠ 汇总（覆盖率陷阱）**
   明细记录数通常只覆盖平台汇总售后单数的一部分（欧莱雅样例：明细 14443 / 汇总 31638 ≈ 46%）。疑因明细只含"班牛工单系统纳管/AI 处理过"的退款单。
   → **退款率必须用汇总口径**（汇总售后单数 ÷ 订单总数），不能用明细记录数当分子（否则严重低估）。同时披露"明细覆盖率"。

## 三、口径硬规则（裁定过的，勿犯）

| 指标 | 正确口径 | 错误做法（踩过的坑） |
|---|---|---|
| **退款率** | 汇总售后单数 ÷ 订单总数（来自汇总文件） | 用明细记录数当分子 → 低估近一半 |
| **负面情绪率** | 负面 ÷ **有情绪记录**（剔除"缺失"） | 缺失计入分母 → 被约半数缺失稀释失真 |
| **各原因占比** | 该原因单数 ÷ **可分析样本**（非全样本） | 用全样本做分母 → 显著性被低信息项淹没 |
| **可分析样本** | 进线≥1 **且** 标签≠非售后场景 | 把静默(零进线)也算进可归因 |

## 四、三级样本体系（呼应 prompt 第 57 条）

报告必须显式呈现，避免"全样本 > 可分析 > 具体分析项"混用导致的不显著：

| 层级 | 定义 | 欧莱雅兰蔻样例 |
|---|---|---|
| 全量 (FULL) | 该品牌全部退款明细 | 14443 |
| 可分析 (ACTIVE) | 进线次数 ≥ 1（有聊天可分析） | 3059 |
| 可归因 (CURRENT) | 进线≥1 且 定位到原因（标签≠非售后场景） | 1201 |

## 五、双层归因（我们独有的洞察）

把"顾客声明原因(平台) × AI 聊天实情(标签)"做成交叉热力图，量化"不想要了"这类低信息退款背后的真实结构（如有进线的"不想要了"里实际多为活动降价/描述不符）。脚本 `dual_layer_crosstab()`。

## 六、显著性标注

- n ≥ 30：正常出结论
- 10 ≤ n < 30：标"样本不足，仅供参考"
- n < 10：标"样本极少，不下结论"，交叉表单元 < 10 不出结论

## 七、典型案例选取（学班牛生产报告）

每个原因取 2–3 条案例，**按情绪烈度排序**（🔴愤怒/投诉 优先），每条含：
`工单编号`（核实锚点）+ 商品名 + SKU + `买家原声`（客户之声）+ `分析依据`（因果推理，数据现成字段）。
→ 证据链翔实是 prompt 的硬要求，`分析依据` 字段是关键。

## 八、新客户调参清单

换客户/品牌时改这几处（其余通用）：
1. 汇总文件 `summary.json`：`{"品牌短名": {"orders": 订单总数, "refund_orders": 平台售后单数}, ...}`。
2. 若品牌自动推断不准（`_brand`），在 `BRAND_CONFIG` 登记精确映射。
3. `LOW_INFO_REASONS`、情绪 emoji 映射、列名——抖音班牛导出通用，一般不用改。
4. 报告里的客户名、Logo、商品举例等表现层内容。

## 九、提案合规护栏（抖音飞鸽/抖店，2026-06 调研核验）

**写提案前必看。把"踩红线/做不到"的措施剔除或改写，否则交付即硬伤。**

### 能做（抖店开放平台有 API 支持，需"售后管理"权限授权过审）
- **售后/退款自动审批**（同意/拒绝）、**物流拦截**、退货入仓确认、**换货**、填物流、举证、超时延长——`/afterSale/operate` 等写接口。
- 订单/物流/售后单据读取；价保是结构化售后类型(`aftersale_type=6`)可经 API 拉取。
- 跨平台背书可用：班牛已作为 ISV 完成**淘系"自助服务大厅"接入**，与店小蜜协同做催物流/改地址/退差价（618 智能解决率 +20%）。

### 做不到（第三方无法主动/自动执行）
- **价保自动退差**：由抖音平台系统在**消费者主动申请后**执行；商家只能后台开通价保服务+承担差价。→ 提案改为"**引导消费者自助申请价保 + 后台开通价保服务**"。
- 淘系退款审批 API 虽存在，但仅子账号授权+权限点+单笔额度上限+两段式短信验证，**不支持无人值守自动审批**（跨平台同理审慎）。

### 红线（高风险，不可作主推）
- **顾客未进线时主动触达消费者做营销/维挽**：商家经飞鸽/短信/微信等联系用户并**引发反感投诉 = 「骚扰他人」违规**（每次约 300 元赔付）；消极服务分级处罚 A2–A12 分、限单/冻结货款，态度差/拉黑扣订单实付 5%(5–30 元)。
  → 改为：**顾客进线后被动应答/引导**，或仅限"订单状态卡片/签收关怀"等平台允许的通知场景，**不做主动营销式触达**。
- "未发货即可秒退"是误读：自动退款取决于实际履约节点(打单/出库/揽收)、金额、商品类型与风控，**自动退款≠秒退**。

> 信源：抖店开放平台官方文档 op.jinritemai.com、抖音电商学习中心「骚扰他人」细则(article 105788)、价保服务规范；淘宝开放平台 open.alitrip.com。**正式客户提案引用具体条款/赔付额前，应再核对官方规则中心当前生效措辞与日期。**

## 十、口径补充（方法范式调研）

- **退款率口径局限**：本 skill 用**日历口径**(期内退款÷期内订单)，因单周快照无法做 cohort。退货滞后正常 14–30 天(Q4 30–60 天)，日历口径会**低估真实退货率**——报告须声明此局限。数据跨度足够(≥1 季度、按发货批次)时应转 **cohort 退款率**(某期发货批次的退款数÷该期发货数)。
- **KPI 主轴建议**：退货率最难控、不宜作首要 KPI；主推**可控的退款率(现金流出占比)+ 单均退货成本(cost-per-return)**，更直接映射毛利。
- 报告结构遵循**金字塔原理**(结论先行、一页纸执行摘要、自上而下表达)——已落实。
