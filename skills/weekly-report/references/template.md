# Template — 周报模板

## 命名约定

```
${KNOWLEDGE_LIB:-~/Knowledge-Library}/05-Reports/weekly/${WK_NUM}-工作周报.md
```

`WK_NUM` 形如 `2026-W19`,等于 ISO 周(`%G-W%V`)。

## Frontmatter

```yaml
---
created: YYYY-MM-DD          # 周报生成日,等于本周周日 / 周六(实际生成日)
updated: YYYY-MM-DD          # 同 created
status: 已完成
project: bytenew-voc
tags:
  - type/周报
  - weekly
  - zentao
  - voc
week: YYYY-WXX               # = WK_NUM
aliases:
  - YYYY-WXX周报
---
```

## 占位变量速查

| 变量 | 来源 |
|---|---|
| `${WK_NUM}` | Setup 阶段计算 |
| `${WK_START_DATE}` / `${WK_END_DATE}` | `WK_START` / `WK_END` 取前 10 字符 |
| `${count_done}` / `${count_progress}` | `jq '[.[]|select(.wk_role=="完成"\|"进行")]|length' /tmp/wk-R1.json` 拆分(完成 vs 进行) |
| `${count_R1}` / `${count_R2}` / `${count_R3}` | `jq 'length' /tmp/wk-RX.json` (R1 已去重父任务) |
| `${NEXT_LABEL}` | 例 `W19`(=WK_NUM 周数 + 1) |
| `${NEXT_S_DATE}` / `${NEXT_E_DATE}` | `NEXT_S` / `NEXT_E` 取前 10 字符 |
| `${GEN_TIME}` | `TZ=Asia/Shanghai date "+%Y-%m-%d %H:%M"` |
| `${NEXT_GEN_DATE}` | 下周六日期 = `mon_ep + 5*86400` 的 +0800 表示 |
| `${WK_NEXT_NUM}` | `${NEXT_S}` 算出的 ISO 周编号 |

## 渲染顺序

LLM 渲染步骤:

1. 用 `/tmp/wk-R1-formatted.json` 渲染 §1.任务列表(完成 / 推进中 两个子段)
2. 写"完成情况说明" — **必读 §叙述风格 §1**,业务归纳风,不出现 T 编号
3. 用 `/tmp/wk-R2.json` + 标题改写后的字符串渲染"已解决"
4. 用 `/tmp/wk-R3.json` 渲染"待跟进"
5. 计算 R2 按 type 分布(见 data-collection.md §Bug 根因分类),填表
6. AI 应用至少 4 行(模板自带 3 行 + 至少补 1 行)
7. §下周计划 — **必读 §叙述风格 §2**,从 R4 + R1 进行中抽,目标列写"动作+成效",加禅道列
8. §向外看:用 ai-daily-digest.md 的输出
9. 保存到 `${KNOWLEDGE_LIB}/05-Reports/weekly/${WK_NUM}-工作周报.md`

## 叙述风格(基于 W15/W16 历史周报提炼,**强约束**)

LLM 渲染 §1 完成情况说明 与 §下周 OKR 时,必须遵守以下风格,否则会被人工驳回。

### 1. 完成情况说明

**做什么**:从 R1 任务名(完成+进行)归纳出**业务主线** + **覆盖范围** + **交付节奏**,3 句以内。把任务名抽象成"业务能力"语言。

**硬约束**:
- ✗ 不出现 T 编号(那是 §1 任务列表的事)
- ✗ 不逐条复述任务做了什么
- ✗ 不出现"doing / done / wait / pause"等 zentao 状态词
- ✗ 不夹带 Bug 内容(那是 §2 的事)
- ✓ 用业务语言:"VOC 评价能力补强"、"链路稳定性保障"、"演示环境部署"
- ✓ 概括数量:"X 个子任务"、"覆盖 A/B/C 模块"
- ✓ 体现节奏判断:"周中 X,周后段 Y"、"主线落在节前清单 + 下周准备"

**正例(W15)**:
> 本周主线是**新版VOC小程序后端开发交付**,6 个子任务全部完成,涵盖商品管理优化、数据源配置、配置/标签场景管理等功能模块,持续推进 voc日常迭代的交付节奏。另完成宝洁回评模型效果优化 1 项。

**正例(W16)**:
> - 本周主线集中在 VOC 链路能力补强与问题排查,一方面推进平台对接和评价情感判断逻辑开发,另一方面补齐演示环境、列表展示和故障排查接口能力。
> - 从节奏上看,周中完成了情感标签与展示相关功能开发,周后段转向接口增强和问题定位,交付重心从单点需求实现延伸到链路稳定性保障。
> - 本周共完成任务 5 个,覆盖演示环境、功能开发、接口增强和对接工作,产出较为聚焦。

**反例(避免)**:
> - 已完成两项后端交付:T43849"618大促VOC评价优化"后端实现于 04-28 节前 finish,...;T43911"评价情感优化针对宝洁环境隔离"于 04-30 节前 finish,...
> - 推进中两项均围绕 T43902 中差评跟踪处理父需求展开:父需求本身在 04-30 完成需求拆解(从 wait → doing);其子任务 T43919...

(为什么是反例:堆 T 编号 + 时间戳 + 状态词,变成了任务流水账,不是周报。)

### 2. 下周 OKR

**做什么**:从 R4(下周待开展)+ R1 进行中任务,挑 3~5 项,按 P0/P1/P2 排。每行有"任务名 / 目标 / 禅道"三列。

**硬约束**:
- ✓ 任务名用业务表达,T 编号放后缀:`中差评跟踪处理/数据源配置 (T43919)`
- ✓ 目标列写"动作 + 成效"句:"启动开发,完成主流程联调"、"修复上线,解决商品名称截断问题"
- ✗ 目标列不要写状态字典:"doing → done"、"从 wait 推到 doing"
- ✓ 禅道列三个标签:**进行中** / **未开始** / **暂无**(本周才发现需要新建的)
- ✓ quote 块前缀注明:"下周主要任务均来自禅道未完成/未开始任务,另补充本周未入禅道的 N 项"(如有补充)
- ✗ 不要硬塞跨 skill 的运维事项(部署、监控)、不要把 bug 收敛延续当成 OKR

**正例(W16)**:
```
> 下周主要任务均来自禅道未完成/未开始任务,另补充两项本周未入禅道的问题修复。

| 优先级 | 任务 | 目标 | 禅道 |
|--------|------|------|------|
| P0 | 宝洁回评生成优化 | 优化宝洁品牌回评生成模型,提升回评准确率与生成质量 | 禅道中暂无(需新建) |
| P1 | 评价列表新增展示商品主图字段(T43538) | 完成前后端联调与上线 | 进行中 |
| P2 | 多个SKU同一商品链接商品卖点获取不准问题 | 修复 SKU 映射逻辑,确保卖点准确归属 | 禅道中暂无(需新建) |
| P2 | 修复评价方案配置模块商品名称为空的问题(T43545) | 定位根因并修复,确保商品名称正确展示 | 未开始 |
```

### 3. 渲染流程修订

LLM 在 §渲染顺序 第 2 步("写完成情况说明")和第 7 步("§下周计划")必须先读本章再写。先草稿 → 自检 3 个硬约束 → 改写 → 提交。如果只能想到状态字典/任务流水账,就**回到 R1/R4 数据,把任务名翻译成业务能力**(例如 T43919"数据源配置" → "中差评数据源接入与配置")。
